ABSTRACT
Ghost Chat is an infrastructure-free, offline peer-to-peer communication system designed and implemented on the Heltec WiFi LoRa 32 V2 development board powered by the ESP32 microcontroller and the SX1276 LoRa radio transceiver. The system operates as a self-contained WiFi access point that hosts a real-time web-based chat application — entirely without internet connectivity, cloud servers, or external infrastructure of any kind.
Using a captive portal architecture, any device connecting to the Ghost Chat access point is automatically redirected to the chat interface through DNS interception. The platform supports three isolated chat rooms (General, Ops, Random), enabling organized communication in diverse operational contexts. Security is enforced through multiple layers: session token authentication prevents unauthorized message injection, per-IP rate limiting blocks message flooding attacks, and backend input sanitization eliminates cross-site scripting (XSS) vulnerabilities. A duress password mechanism allows a one-touch silent wipe of all data and SPIFFS storage.
The LoRa SX1276 radio module extends the system's reach beyond WiFi range by enabling message relay between multiple Ghost Chat nodes operating within 500m to 2km in urban environments and up to 5km in open terrain at the legal Indian frequency of 866 MHz (IN865 band). Additional features include self-destruct messages with client-side countdown timers, view-once image transfer with automatic deletion, message reactions, typing indicators, pinned messages, and an OLED display showing live node status.
The entire application stack — firmware, HTML5 interface, CSS, and JavaScript — runs directly on the microcontroller's flash memory. No external computer, server, or internet connection is required at any point. This makes Ghost Chat particularly suitable for deployment in disaster response, field operations, campus environments, cybersecurity research, and any scenario where existing communication infrastructure is unavailable, compromised, or untrusted.

1.  INTRODUCTION
The digital age has made human beings almost entirely dependent on centralized communication infrastructure — mobile networks, cloud servers, and internet backbones — for everyday interaction. While this centralization delivers convenience and scale, it creates critical single points of failure. Natural disasters, power outages, network congestion, censorship, and deliberate infrastructure attacks can sever communication at the worst possible moments.
Beyond availability, centralized systems raise fundamental privacy and surveillance concerns. Every message transmitted through commercial platforms passes through servers that may log, analyze, or share data with third parties. In campus environments, organizational settings, or field operations, there is often a need for communication channels that are both reliable and auditable only by the parties involved.
Ghost Chat addresses these challenges by creating a fully self-contained communication node using commodity embedded hardware. The Heltec WiFi LoRa 32 V2 development board — built around the ESP32 dual-core processor and equipped with an SX1276 LoRa radio transceiver — serves as both the WiFi access point and the web server. Users connect to the Ghost Chat network exactly as they would connect to any WiFi hotspot, and a captive portal mechanism automatically presents the chat interface without any app installation.
The system is built entirely in C++ (Arduino framework) with the web interface embedded directly in the microcontroller's flash memory as a PROGMEM string. This architecture eliminates all external dependencies: no cloud, no router, no internet, no app store. Ghost Chat can be operational in under 30 seconds from power-on, anywhere in the world, on a single USB power bank.
1.1 Key Contributions
Design and implementation of a complete offline chat system on a microcontroller platform.
Multi-layer security architecture: session tokens, rate limiting, XSS sanitization, and duress wipe.
LoRa mesh relay extending communication range beyond WiFi coverage using the legal IN865 band.
Self-contained captive portal architecture supporting iOS, Android, and Windows automatic redirect.
View-once (ephemeral) image transfer with automatic server-side deletion.
Multi-room chat architecture with isolated message histories for different communication contexts.
2.  PROBLEM STATEMENT
Despite the proliferation of messaging applications, several fundamental gaps remain unaddressed in current communication solutions:
2.1 Infrastructure Dependency
All mainstream messaging platforms — WhatsApp, Telegram, Signal, iMessage — require active internet connectivity and remote servers. In disaster scenarios, remote field environments, or network-denied areas, these tools cease to function entirely. There is no widely available solution that provides full-featured real-time chat without any external infrastructure.
2.2 Privacy and Surveillance Concerns
Centralized platforms maintain server-side logs even when end-to-end encryption is employed at the message layer. Metadata — who communicated with whom, when, and how often — remains visible to platform operators and potentially to law enforcement or adversaries. Groups requiring truly ephemeral, unlogged communication have no good solution.
2.3 Installation and Onboarding Friction
Deploying a new communication tool typically requires app installation, account creation, phone number verification, and ongoing software updates. In emergency or ad-hoc scenarios, this friction is unacceptable. A zero-installation, browser-based solution accessible from any WiFi-capable device is needed.
2.4 Range Limitations of WiFi
WiFi access points provide coverage of approximately 30–50 meters indoors and up to 100 meters in open areas. For campus-wide or field deployment requiring communication across buildings or terrain, WiFi alone is insufficient. A low-power, long-range relay mechanism is required to extend the network without additional infrastructure.
2.5 Lack of Security in Embedded Chat Systems
Existing DIY embedded chat implementations found in hobbyist communities lack even basic security measures — no authentication, no rate limiting, no XSS protection, and no mechanism to wipe data under duress. These systems are unsuitable for real operational use.

5.  SYSTEM ARCHITECTURE
Ghost Chat employs a single-node embedded server architecture in which the ESP32 microcontroller simultaneously acts as the WiFi access point, DNS server, HTTP web server, file storage system, LoRa transceiver, and OLED display controller. The architecture is designed for maximum self-containment with zero external dependencies.
5.1 High-Level Architecture
The system operates across four primary layers:
Layer
Components
Physical Layer
ESP32 hardware, SX1276 LoRa chip, SSD1306 OLED display, SPIFFS flash storage
Network Layer
WiFi AP (802.11 b/g/n) + DNS server intercepting all domain queries
Application Layer
HTTP WebServer on port 80 serving HTML/CSS/JS and RESTful API endpoints
Presentation Layer
Browser-based HTML5 chat UI — zero installation, works on all modern browsers
5.2 Request Flow
The following sequence describes the complete flow from device connection to message delivery:
Step 1: Client device connects to Ghost Chat WiFi (SSID: Ghost_Net)
Step 2: Device OS detects captive portal via HTTP connectivity probes
Step 3: ESP32 DNS server responds to ALL domain queries with its own IP (192.168.4.1)
Step 4: OS or user opens browser → HTTP GET / → ESP32 serves INDEX_HTML from PROGMEM
Step 5: User enters alias and room password → POST /login → session token created → cookie set
Step 6: Chat polls /read every 1.5 seconds → receives messages + typing indicators
Step 7: User sends message → POST /send (rate-checked, token-validated) → stored in rooms[]
Step 8: LoRa broadcast → nearby Ghost Chat nodes receive relay message → inject into General room
Step 9: Outgoing message XOR-encrypted in browser with room key before transmission
5.3 Multi-Room Architecture
The server maintains three independent String buffers (rooms[0], rooms[1], rooms[2]) in SRAM, each capped at 28,000 bytes. Messages are stored as: ENCODED_PAYLOAD|||. The room index is passed as a URL parameter (?room=N) with all API calls, ensuring complete isolation between the General, Ops, and Random rooms. Pinned messages and typing indicators are global (shared across the active session), but room content is strictly separated.
5.4 Security Architecture
Ghost Chat implements defense-in-depth with four independent security mechanisms:
Mechanism
Description
Session Tokens
8-char hex token generated per login via esp_random(). Validated on every write request.
Rate Limiting
Per-IP message counter: max 10 messages per 60-second window. Returns HTTP 429 on violation.
XSS Sanitization
All stored message content is scanned for <, >, &, ', " characters and escaped before storage.
Duress Wipe
A configurable duress password triggers silent format of SPIFFS and clearing of all RAM buffers.
Path Traversal Guard
All file operation URLs are sanitized to strip ../ and / prefix injection.
Panic Button
Physical GPIO0 button: 3-second hold wipes everything and shows confirmation on OLED.
6.  TECHNOLOGY STACK
Ghost Chat is implemented using a carefully selected stack of embedded and web technologies, all chosen for their proven stability, low resource consumption, and compatibility with the ESP32 platform.
6.1 Firmware Technologies
C++ (Arduino Framework)
The core firmware is written in C++ using the Arduino framework, which provides hardware abstraction for the ESP32's WiFi, SPI, I2C, GPIO, and SPIFFS peripherals. The Arduino framework enables rapid development while retaining direct access to ESP-IDF functions such as esp_random() for cryptographically suitable random number generation.
ESP-IDF (via Arduino Core)
The underlying Espressif IoT Development Framework provides the FreeRTOS kernel, WiFi stack, and hardware abstraction. The Arduino ESP32 core (v2.x) wraps these components and exposes familiar Arduino APIs while supporting advanced features like multitasking, PROGMEM for flash-stored constant data, and the U8g2 graphics library.
6.2 Web Interface Technologies
HTML5 / CSS3
The entire chat UI is a single-file HTML5 application stored as a PROGMEM constant in flash memory. CSS3 features including backdrop-filter, CSS custom properties (variables), CSS animations, and flexible layout (Flexbox) are used extensively to produce a modern, responsive interface without any external stylesheet dependencies.
Vanilla JavaScript (ES6+)
The client-side logic is implemented in pure JavaScript without any frameworks or libraries. Key APIs used: Fetch API for HTTP polling and message submission, Web Audio API for sound notifications, FileReader and Canvas API for client-side image compression before upload, and the XOR cipher implemented in JavaScript for message encoding on the client side.

7.4 Critical V2 Hardware Notes
OLED RST (GPIO16) MUST be pulled LOW then HIGH in setup() or the display remains blank. This is the most common source of display failure on V2 boards.
SPI bus MUST be manually initialized with SPI.begin(5, 19, 27, 18) before any LoRa commands. The V2 does not use ESP32 default SPI pins.
The LoRa library (Sandeep Mistry) takes frequency in Hz: 866E6. RadioLib (for SX1262/V3) takes MHz: 866.0. Mixing these causes complete radio failure.
SPIFFS partition must be set to 'Default 4MB with spiffs' in Arduino IDE before flashing. Without this, SPIFFS.begin() always fails.
GPIO2 is a boot-strapping pin on ESP32. Avoid using it as LED or output during startup.
8.  MODULE DESCRIPTION
Ghost Chat is composed of ten tightly integrated software modules running on the ESP32 firmware, each serving a distinct functional role.
8.1 WiFi Access Point Module
Creates and manages the ESP32 software access point that clients connect to.
Configures ESP32 in WIFI_AP mode with a custom SSID (Ghost_Net) and password
Supports up to 8 simultaneous connected devices (ESP32 AP limit)
Provides WiFi.softAPgetStationNum() for real-time connected device count
IP assignment via built-in DHCP server (192.168.4.x subnet)
Survives client disconnect/reconnect without requiring board reboot
8.2 Captive Portal / DNS Module
Intercepts all DNS queries from connected clients and redirects them to the chat interface.
DNSServer listens on UDP port 53 and responds to ALL domain queries with 192.168.4.1
iOS captive portal: handles /hotspot-detect.html and /library/test/success.html
Android captive portal: handles /generate_204 and /gen_204
Windows NCSI: handles /connecttest.txt with 'Microsoft Connect Test' response
404 catch-all: all unrecognized paths redirect to http://ghost.chat (domain name)
8.3 Authentication Module
Manages user login, session creation, and request authorization.
Login endpoint (/login) validates room password against stored ROOM_PASS constant
On success: creates session entry with IP + 8-char cryptographically random token
Session token returned to client and used as ?tok= parameter in all write requests
Duress password check: matches DURESS_PASS → triggers doWipe() silently, returns 'WIPED'
Token validation on /send, /typing, /pin, /clear, /delete_file endpoints
Sessions persist in SRAM — cleared on wipe or reboot (no persistent auth)
8.4 Rate Limiter Module
Prevents message flooding by enforcing per-IP message rate limits.
Tracks up to 16 unique IP addresses in a fixed-size SRAM array
Per-IP counter reset every 60 seconds (sliding window)
Maximum 10 messages per IP per 60-second window
Returns HTTP 429 (Too Many Requests) when limit exceeded
IP removed from tracking when counter resets to allow new window
8.5 Chat / Messaging Module
Core real-time messaging engine with multi-room support.
Three independent String buffers: rooms[0] (General), rooms[1] (Ops), rooms[2] (Random)
Messages stored as: ENCODED_PAYLOAD||| (triple-pipe delimiter for easy parsing)
Auto-trim: each room capped at 28,000 bytes, oldest messages discarded first
Client-side XOR cipher: messages encrypted with room key before transmission
Typing indicator: /typing endpoint stores active typer + room, expires in 3 seconds
Poll interval: 1,500ms — clients fetch /read?room=N to receive messages
Self-destruct: SD:duration:epoch:content prefix preserved in storage; client filters expired
8.6 File Storage Module (SPIFFS)
Manages image upload, download, and deletion using flash filesystem.
Images compressed to JPEG quality 0.65 and max 560px dimension on client before upload
Upload handler: UPLOAD_FILE_START → UPLOAD_FILE_WRITE → UPLOAD_FILE_END (race-condition-free)
Download: streams file to client with image/jpeg content type
View-once (ephemeral) images: deleted immediately when burnS() is triggered by client
Path traversal protection: all filenames sanitized to strip ../ and / prefix
File deletion tracked via CMD:DEL: messages in chat history for all-client sync
8.7 Security & Wipe Module
Implements duress wipe, panic button, and data sanitization.
Duress password: silently calls doWipe() — clears all rooms[], sessions[], rateTable[], formats SPIFFS
Physical panic button (GPIO0): 3-second hold triggers wipe with OLED countdown display
Backend XSS sanitizer: scans all /send content for <, >, &, ', " and escapes before storage
Message length cap: 480 characters enforced in sanitize(), 1500 bytes enforced in /send handler
Admin wipe: POST /admin_wipe with admin password triggers full wipe remotely
8.8 LoRa Relay Module
Broadcasts outgoing messages via SX1276 LoRa radio and receives relay messages from nearby nodes.
Frequency: 866 MHz (center of India's license-free IN865 band: 865–867 MHz)
Spreading Factor 7, Bandwidth 125 kHz, Coding Rate 4/5, TX Power 17 dBm
Custom sync word 0xAB ensures only Ghost Chat nodes receive each other's transmissions
loraBroadcast(): called on every /send — transmits encoded message payload (max 200 bytes)
loraReceive(): called every loop iteration — non-blocking parsePacket() check
Self-filtering: messages from own NODE_ID are discarded to prevent echo loops
Relay injection: received LoRa messages added to rooms[0] (General) as system messages
OLED displays LoRa RSSI of last received packet for signal quality monitoring
8.9 OLED Display Module
Provides real-time node status on the 0.96-inch SSD1306 display.
Updates every 3 seconds to avoid I2C bus contention with main loop
Displays: Node name, IP address, online user count, free heap RAM, LoRa status + RSSI
Boot sequence: shows startup animation through WiFi, SPIFFS, and LoRa initialization phases
Panic wipe sequence: countdown display (3…2…1) and confirmation screen
CRITICAL: GPIO16 must be toggled LOW→HIGH in setup() before u8g2.begin() or display stays blank
8.10 Admin Module
Provides operator access to node diagnostics and remote management.
GET /admin?pass=ADMINPASS: returns JSON with RAM, uptime, sessions, LoRa RSSI, room sizes
POST /admin_wipe?pass=ADMINPASS: triggers full data wipe from any connected client
GET /status: returns plain-text health summary (RAM, uptime, WiFi stations, LoRa, Node ID)
GET /ram: returns free heap in KB — polled by client to show/hide low-RAM warning banner
GET /lorastatus: returns OK:RSSI:seconds_since_last_rx or OFF if LoRa disabled

10.  IMPLEMENTATION DETAILS
This section describes the key implementation techniques and code structures used in Ghost Chat V3.
10.1 Session Token Generation
Session tokens are generated using the ESP32's hardware random number generator via the esp_random() function, which sources entropy from RF noise and hardware. The 32-bit random value is formatted as an 8-character hexadecimal string using snprintf(). This provides 2^32 (4.29 billion) possible tokens — sufficient for the 8-client maximum of the access point.
Token validation on every write request:
bool validateTok(uint32_t ip, const String& tok) {
  for (int i = 0; i < sessionCount; i++) {
    if (sessions[i].ip == ip && tok == String(sessions[i].tok)) {
      sessions[i].lastSeen = millis();
      return true;
    }
  }
  return false;
}
10.2 XOR Cipher (Client-Side)
Messages are XOR-encoded in the browser before transmission using the room password as the key. This is not cryptographically strong encryption — it prevents casual interception by passive WiFi sniffers but is not resistant to known-plaintext attacks. The implementation uses Base64 encoding of the XOR result to produce ASCII-safe transmission. True end-to-end encryption using ECDH is planned for V4.
10.3 RAM Management
The ESP32 classic (ESP32-WROOM-32) has approximately 520 KB of SRAM, of which roughly 300 KB is available to the application after the WiFi stack, FreeRTOS kernel, and TCP/IP buffers claim their share. Each of the three chat rooms is capped at 28,000 bytes, and the minimum free heap threshold is set at 28,000 bytes. When free heap drops below this threshold, trimRoom() is called on all three rooms to discard the oldest 20% of messages.
10.4 LoRa Configuration and Relay
The LoRa relay is initialized in setup() after SPI.begin():
SPI.begin(LORA_SCK, LORA_MISO, LORA_MOSI, LORA_SS);
LoRa.setPins(LORA_SS, LORA_RST, LORA_DIO0);
if (LoRa.begin(866E6)) {                  // 866 MHz
  LoRa.setSpreadingFactor(7);
  LoRa.setSignalBandwidth(125E3);
  LoRa.setCodingRate4(5);
  LoRa.setTxPower(17);
  LoRa.setSyncWord(0xAB);  // Ghost Chat only
  LoRa.receive();
  loraOK = true;
}
The custom sync word 0xAB ensures that the SX1276 hardware-filters all LoRa packets not originating from Ghost Chat nodes, reducing interrupt load. loraReceive() is called every loop iteration and uses the non-blocking LoRa.parsePacket() — it returns immediately if no packet is waiting, consuming negligible CPU time.

10.6 Image Compression Pipeline
Before uploading, images are compressed entirely in the browser using Canvas API — no server-side processing required. The pipeline: FileReader reads the file as DataURL → Image object decodes it → Canvas resizes to max 560px while preserving aspect ratio → Canvas.toBlob() encodes at JPEG quality 0.65 → FormData posts the blob to /upload. This typically reduces a 3MB smartphone photo to under 80KB, well within SPIFFS capacity.
10.7 Self-Destruct Messages
Self-destruct timers are implemented entirely on the client side. When a user selects a timer and sends a message, the content is prefixed with SD:duration:epoch: where duration is the timer in seconds and epoch is Date.now() (milliseconds since Unix epoch). All receiving clients parse this prefix and compute the remaining time as: remaining = duration - floor((Date.now() - epoch) / 1000). Messages with remaining ≤ 0 are not rendered and instead show a '💀 expired' system line. The server stores the prefixed content verbatim — it has no knowledge of timer state.

11.1 Performance Summary
The system demonstrates stable operation under realistic multi-user load. Memory consumption remains well above the 28 KB safety threshold even at maximum message storage capacity. The 1.5-second polling interval provides acceptable real-time feel while avoiding excessive HTTP request load on the ESP32's single-threaded server. LoRa relay adds negligible latency (< 200ms for the LoRa transmission itself) to message delivery.
12.  LIMITATIONS
1. WiFi Range Constraint
The ESP32 access point provides reliable coverage of approximately 30–50 meters indoors. Beyond this range, clients cannot connect. LoRa relay partially mitigates this for node-to-node relay but not for direct client connection.
2. Maximum 8 Concurrent Users
The ESP32 WiFi hardware limits the access point to 8 simultaneously connected stations. This is sufficient for small group communication but not suitable for large gatherings.
3. No True End-to-End Encryption
The XOR cipher used for message encoding is not cryptographically secure. A passive attacker who knows the room password can trivially decode all messages. True E2E encryption (ECDH + AES) is planned but not yet implemented.
4. SPIFFS Storage Limit
The 2 MB SPIFFS partition limits image storage to approximately 20–30 compressed images before the filesystem fills up. LittleFS migration in V4 will provide better wear-leveling and slightly more usable space.
5. Single Point of Failure
The current architecture is a star topology — all communication routes through the single ESP32 node. If the node loses power, all communication stops. The planned mesh topology will distribute this risk across multiple nodes.
6. No Message Persistence Across Reboots
All message history is stored in SRAM and lost on reboot. SPIFFS persistence for chat history would require careful serialization and is not implemented in the current version.
7. Polling-Based Real-Time Communication
The 1.5-second HTTP polling interval introduces noticeable message delivery latency compared to WebSocket-based systems. The ESP32's WebServer library does not natively support WebSockets; a custom WebSocket implementation would improve real-time feel.
8. LoRa Single-Hop Only
The current LoRa relay is broadcast-based and single-hop: Node A transmits, Node B receives and injects into its General room. True mesh routing with multi-hop forwarding is not yet implemented.
13.  FUTURE ENHANCEMENTS
The following enhancements are planned for Ghost Chat V4 and beyond, building on the solid foundation of the current V3 implementation:
13.1 True LoRa Mesh Networking
Implement a multi-hop routing protocol (similar to Meshtastic's flood routing) allowing messages to hop across a chain of Ghost Chat nodes. A node ID table and hop count limit (TTL) will prevent broadcast storms. This could extend network coverage to several kilometers using 3–4 nodes positioned strategically.
13.2 End-to-End Encryption (ECDH + AES-128)
Replace the current XOR cipher with proper cryptography: Elliptic Curve Diffie-Hellman (ECDH) key exchange on login, followed by AES-128 in CTR mode for message encryption. The ESP32 hardware AES accelerator makes this feasible without significant performance impact.
13.3 LittleFS Migration
Replace SPIFFS with LittleFS for improved flash wear-leveling, support for directories, and slightly better write performance. The migration is straightforward — Arduino ESP32 core includes LittleFS and the API is nearly identical to SPIFFS.
13.4 Voice Messages
Implement 10–15 second compressed voice note recording using the Web Audio API's MediaRecorder with Opus codec. The compressed audio (typically 15–30 KB) would be uploaded to SPIFFS and shared as a playable audio bubble in the chat.
13.5 Battery-Optimized Field Mode
Implement a field mode that reduces WiFi TX power, increases polling interval to 5 seconds, and dims the OLED display. ESP32 modem sleep between poll cycles could extend battery life from a 2000mAh LiPo to 6–8 hours of continuous operation.
13.6 GPS Location Sharing
Integrate a Neo-6M GPS module via UART to enable location sharing as a special message type. Recipients would see a coordinate pair that can be copied into offline map applications.
13.7 Android Companion App
A lightweight Android WebView wrapper for Ghost Chat that provides push notifications when new messages arrive, bypassing the need for the captive portal auto-redirect mechanism and providing a more native app experience.
14.  CONCLUSION
Ghost Chat demonstrates that a complete, secure, multi-featured real-time communication system can be built and operated entirely at the hardware edge — without cloud infrastructure, without internet connectivity, and without any software installation on client devices. The project validates the viability of the ESP32 microcontroller as a communication platform for scenarios where existing infrastructure is unavailable or untrusted.
The V3 implementation achieves all primary objectives: a working captive portal supporting iOS, Android, and Windows automatic redirect; multi-room isolated chat; a four-layer security architecture including session authentication, rate limiting, XSS sanitization, and duress wipe; and LoRa radio relay extending the network's reach beyond WiFi range at the legally compliant Indian frequency of 866 MHz.
The project also revealed important engineering challenges inherent to embedded web application development: strict memory budget management across three concurrent SRAM chat buffers, the complexity of cross-platform captive portal compatibility, and the tradeoffs between polling latency and server load on single-threaded microcontroller HTTP servers. Each of these challenges was addressed through careful design decisions documented in this report.
Ghost Chat has practical applications in disaster response communications, campus-scale private networks, cybersecurity research into captive portal and LoRa mesh architectures, and any environment where surveillance-free, infrastructure-free communication is required. The planned V4 enhancements — true mesh networking, ECDH/AES encryption, and voice messaging — will further expand the system's operational capabilities.
This project demonstrates that meaningful security and privacy tools can emerge from accessible, affordable hardware and open-source software ecosystems — a principle that aligns with the broader goals of democratic access to secure communication technology.
